# Lavender Design System

---

## The Problem

Building financial products means building interfaces that handle complexity — dense data tables, real-time charts, multi-step forms, theme preferences, accessibility requirements. Teams run into the same friction every time:

- **Inconsistency at scale.** Without a shared component library, every feature team builds its own buttons, modals, and form inputs. Visual drift compounds across screens until the product feels like a patchwork of different apps.
- **Accessibility as an afterthought.** Keyboard navigation, focus management, ARIA attributes, and color contrast get skipped under deadline pressure, then become expensive to retrofit.
- **Theming is brittle.** Hardcoded colors scatter across files. Adding a dark mode means touching every component. Maintaining two themes in parallel becomes a full-time job.
- **No single source of truth.** Designers reference Figma, developers reference code, PMs reference the live app — and all three disagree on what the current state of the UI actually is.

---

## The Solution

Lavender is a **design system built for scalable financial interfaces**. It provides 25 production-ready React components backed by a token-driven theme system, strict TypeScript typing, and living documentation.

### Token-driven theming

Every color, radius, and spacing value flows through CSS custom properties. Components never hardcode visual values — they reference semantic tokens (`--primary`, `--border`, `--muted`) that resolve differently per theme. Switching between Lavender Dawn and Lavender Moon is a single class toggle with zero runtime cost.

### Accessibility by default

Keyboard navigation, ARIA attributes, and focus management are built into every interactive component — not bolted on later. Accordion, Tabs, Modal, Tooltip, and Dropdown all support full keyboard control out of the box. Color contrast ratios meet WCAG AA in both themes.

### Type-safe component contracts

Every component has a dedicated TypeScript interface. Generic type parameters on data-driven components (Autocomplete, VirtualizedList, Dropdown) enforce type safety from the API boundary through to the render layer. No `any` usage anywhere in the codebase.

### Living documentation

This Storybook is the single source of truth. Every component has a Playground story wired to controls, feature stories covering each variant and state, and interaction tests that run headlessly in CI. What you see here is what ships.
