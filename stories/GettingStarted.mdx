# Getting Started

Everything you need to set up a development environment and work in the Lavender Design System codebase.

---

## Environment Setup

```bash
git clone <repo-url>
cd lavender-storybook
npm install
npm run storybook
```

Storybook launches at [http://localhost:6006](http://localhost:6006). Changes to components and stories hot-reload automatically.

```bash
npm install clsx tailwind-merge lucide-react tailwindcss-animate
npm install recharts  # only if using Chart components
```

| `clsx` + `tailwind-merge` | The `cn()` class-merging utility used by every component |

| `tailwindcss-animate` | Accordion expand/collapse and spinner animations |
| `recharts` | Only needed for Chart components |

## The `cn()` Utility

Create `lib/utils.ts`:

```ts
import { type ClassValue, clsx } from "clsx"
import { twMerge } from "tailwind-merge"

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

Every component uses `cn()` for class merging. It combines `clsx` (conditional classes) with `tailwind-merge` (deduplicates conflicting Tailwind utilities).

## ThemeProvider

Wrap your app to enable theme switching with localStorage persistence:

```tsx
import { ThemeProvider } from "@/components/ui/ThemeProvider"

function App() {
  return (
    <ThemeProvider defaultTheme="light" storageKey="lavender-theme">
      {/* your app */}
    </ThemeProvider>
  )
}
```

Toggle programmatically with the `useTheme` hook:

```tsx
import { useTheme } from "@/components/ui/ThemeProvider"

const { theme, setTheme } = useTheme()
setTheme(theme === "light" ? "dark" : "light")
```

Switching is controlled by the `dark` class on the document root:

```html
<html>              <!-- Lavender Dawn -->
<html class="dark"> <!-- Lavender Moon -->
```

## TypeScript Conventions

Every component is strictly typed. No `any` usage is permitted.

### File layout

Each component has a dedicated types file:

```
src/components/ui/Button/
├── Button.tsx           # Implementation
├── Button.types.ts      # Props interfaces and type aliases
├── Button.styles.ts     # Variant/size class maps (optional)
└── index.ts             # Barrel exports (components + types)
```

### Prop typing

Components that wrap a single DOM element extend the corresponding HTML attributes:

```tsx
// Button.types.ts
export type ButtonVariant = "default" | "secondary" | "destructive" | "outline" | "ghost" | "link"
export type ButtonSize = "sm" | "default" | "lg" | "icon"

export interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: ButtonVariant
  size?: ButtonSize
}
```

Data-driven components use generics:

```tsx
// Autocomplete.types.ts
export type AutocompleteProps<T> = {
  fetchSuggestions: (query: string) => Promise<T[]>
  getOptionLabel: (option: T) => string
  onSelect: (option: T) => void
  placeholder?: string
  className?: string
}
```

### Ref forwarding

All DOM-wrapping components use `React.forwardRef` with explicit generics:

```tsx
export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>(
  ({ className, variant = "default", size = "default", ...props }, ref) => {
    return <button ref={ref} className={cn(/* ... */)} {...props} />
  },
)
Button.displayName = "Button"
```

### Barrel exports

Every `index.ts` exports both components and types:

```tsx
export { Button } from "./Button"
export type { ButtonVariant, ButtonSize, ButtonProps } from "./Button.types"
```

### Strict rules

| Rule | Rationale |
|------|-----------|
| No `any` | Use `unknown` + type guards, or use specific library types |
| No implicit `any` | Strict mode is on — all parameters must be typed |
| No `@ts-ignore` | Use `@ts-expect-error` with a comment if needed |
| `import type` for type-only imports | Prevents runtime import of type-only modules |
| String literal unions over enums | Lighter, better Storybook controls integration |

## Testing

The project uses **Storybook interaction tests** — `play` functions that run in a real browser via Playwright.

### Setup

Install Playwright browsers (one-time):

```bash
npx playwright install chromium
```

### Commands

Storybook must be running before executing tests.

| Script | Description |
|--------|-------------|
| `npm run test-storybook` | Run all interaction tests against `http://localhost:6006` |
| `npm run test-storybook:ci` | CI mode — exits after a single run |
| `npm run test-storybook:coverage` | Run tests with code coverage |

Filter by file:

```bash
npm run test-storybook -- --testPathPattern="button"
```

### Viewing in the browser

1. Navigate to any story in **Canvas** view (not Docs)
2. Open the **Interactions** panel at the bottom
3. Each `play` function step is listed with pass/fail status

### Accessibility checks

The `@storybook/addon-a11y` addon runs axe-core on every story. Open the **Accessibility** panel to see results. Enforced rules: `color-contrast`, `label`, `button-name`, `aria-roles`, `aria-valid-attr-value`.
